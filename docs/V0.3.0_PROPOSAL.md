# DevMan v0.3.0 中期扩展设计方案

> 文档更新时间：2026-05-23
> 目标版本：v0.3.0
> 版本定位：开发环境配置与自动维护中心

---

## 一、版本目标

v0.3.0 的重点不再是继续增加扫描器，而是把 DevMan 从“发现环境、展示空间、手动清理”的工具，推进到“可配置、可回滚、可提醒、可辅助安装”的开发环境管理中心。

核心目标：

1. **操作可预览**：所有会修改系统或工具配置的操作，执行前必须展示影响范围。
2. **配置可回滚**：网络镜像、代理、安装器选择、自动清理策略等都必须有历史记录和恢复能力。
3. **任务可追踪**：安装、配置、清理、扫描等长任务应记录进度和结果。
4. **体验更高效**：快捷键、趋势图和提醒机制减少重复手动操作。
5. **安全优先**：v0.3.0 允许写配置、执行安装命令，但必须使用白名单、确认弹窗和后端校验。

---

## 二、功能范围

### 1. 快捷键支持

现状：

- `GlobalSearch` 已支持 `Ctrl+K`。
- `App.tsx` 维护 `currentPage`，适合集中处理页面跳转快捷键。
- Dashboard、Cleaner、Migration 等页面已有刷新或执行入口，但没有统一快捷键约定。

方案：

- 新增统一快捷键注册层，先放在前端即可。
- 支持：
  - `Ctrl+K`：打开全局搜索，保留现有行为。
  - `Ctrl+R`：当前页面刷新或重新分析。
  - `Ctrl+1..6`：切换 Dashboard / Environments / Migration / Cleaner / Versions / Settings。
  - `Esc`：关闭搜索、确认弹窗或返回安全状态。
- 页面级动作通过轻量 action registry 注册，不直接在全局层调用页面内部实现。

成功标准：

- 用户可通过键盘完成主要导航。
- 快捷键不会在输入框、文本框中误触发破坏输入。
- `Ctrl+K` 现有搜索体验不回退。

---

### 2. 数据可视化图表

现状：

- Dashboard 已展示当前磁盘和环境空间排行。
- `history` 表记录操作，但缺少周期性空间采样。
- Cleaner 可计算可清理项大小，但不会保存趋势。

方案：

新增轻量采样模型：

```go
type MetricSnapshot struct {
    Id          int64  `json:"Id"`
    MetricKey   string `json:"MetricKey"`
    TargetKey   string `json:"TargetKey"`
    ValueBytes  int64  `json:"ValueBytes"`
    CapturedAt  string `json:"CapturedAt"`
}
```

建议先采集：

- C 盘使用量。
- 各环境 `TotalSize`。
- Cleaner 可释放总空间。
- 每次清理释放空间。

前端第一版不强依赖复杂图表库，可先实现：

- Dashboard 趋势卡片。
- 最近 7 次扫描的空间变化折线。
- Cleaner 节省空间累计值。

成功标准：

- 扫描或清理后生成采样记录。
- Dashboard 能展示空间趋势，而不是只展示当前值。
- 历史数据增长有上限策略，例如只保留最近 180 天或最近 1000 条。

### 3. 自动清理策略

现状：

- Cleaner 已能识别缓存类、依赖类和构建产物。
- 后端 `CleanItems` 已做白名单和安全路径校验。
- Settings 已持久化，但没有清理策略字段。

方案：

v0.3.0 只做“自动分析 + 提醒”，不做后台无确认删除。

新增设置项：

```go
type CleanPolicySettings struct {
    Enabled          bool  `json:"Enabled"`
    AnalyzeOnStartup bool  `json:"AnalyzeOnStartup"`
    WarnAboveBytes   int64 `json:"WarnAboveBytes"`
    IncludeDeps      bool  `json:"IncludeDeps"`
}
```

行为：

- 启动时或手动刷新后自动分析可清理项。
- 当可释放空间超过阈值时，Toast + Dashboard 提醒。
- 用户点击提醒进入 Cleaner，仍需手动确认清理。
- 默认不包含 `node_modules` / `target` 等项目依赖或构建产物。

成功标准：

- 不发生静默删除。
- 提醒可关闭或禁用。
- 清理执行仍复用现有 Confirm 和后端白名单。

### 4. 网络代理与镜像源统一配置

现状：

- Settings 已可保存配置，但没有工具配置写入能力。
- Scanner 可识别 npm/pnpm/yarn/bun、Python、Go、Rust 等工具。
- 修改镜像源会影响用户全局开发环境，必须可回滚。

方案：

第一版优先做“镜像源配置”，代理配置作为后续扩展。

覆盖工具：

| 工具 | 配置内容 | 建议方式 |
|---|---|---|
| npm/pnpm/yarn | registry / proxy / https-proxy | 管理用户级配置，记录 `.npmrc` 或等效命令读出的原值 |
| pip | index-url / extra-index-url / proxy / trusted-host | 管理用户级 pip 配置文件或命令 |
| cargo | registry / source replacement | 管理 `%USERPROFILE%\.cargo\config.toml` |
| Go | GOPROXY / GONOSUMDB | 管理 `go env -w`，并记录原值 |

关键设计：

- 新增 `ConfigProfile`，例如“官方源”“国内镜像”“公司代理”。
- 执行前展示变更预览。
- 执行前备份原配置。
- 提供“一键恢复上次配置”。
- 所有写入进入 `history`。

调研补充：

- npm 存在内置、全局、用户、项目多级 `.npmrc`，v0.3.0 只建议管理用户级配置，避免改项目仓库。
- pip 可通过用户级配置文件、环境变量或 `pip config` 管理，DevMan 应避免写入虚拟环境内配置。
- Cargo 镜像通常通过 `%USERPROFILE%\.cargo\config.toml` 的 `[source]` replacement 实现，写入前必须保留完整原文件。
- Go 推荐通过 `go env -w GOPROXY=...` 写入用户 Go 环境配置，恢复时应写回原值而不是简单清空。
- 环境变量如 `HTTP_PROXY` / `HTTPS_PROXY` / `NO_PROXY` 会影响多个工具，v0.3.0 不建议默认修改系统环境变量。

成功标准：

- 用户可以看到每个工具将被改成什么。
- 配置失败能显示错误且不影响其他工具继续回滚。
- 至少 npm / pip / cargo / go 支持读取当前配置和恢复。

### 5. 一键安装环境

现状：

- DevMan 已能扫描环境，但不能安装环境。
- Windows 目标用户可能同时使用 winget、scoop、choco。
- 安装命令执行有较高风险，必须白名单化。

方案：

v0.3.0 先做“安装中心”，不做复杂依赖编排。

后端新增 installer 抽象：

```go
type PackageManager string

const (
    PackageManagerWinget PackageManager = "winget"
    PackageManagerScoop  PackageManager = "scoop"
    PackageManagerChoco  PackageManager = "choco"
)

type InstallablePackage struct {
    Key          string         `json:"Key"`
    Name         string         `json:"Name"`
    Manager      PackageManager `json:"Manager"`
    PackageId    string         `json:"PackageId"`
    Description  string         `json:"Description"`
    Category     string         `json:"Category"`
}
```

安装流程：

1. 检测 winget / scoop / choco 是否可用。
2. 展示 DevMan 内置白名单包。
3. 用户选择包和安装器。
4. 后端生成命令预览。
5. Confirm 二次确认。
6. 执行安装并 emit 进度事件。
7. 安装完成后自动 `ScanAll()`。

白名单原则：

- 前端不能传任意命令。
- 后端只接受内置 `PackageKey`。
- `PackageId` 来自后端白名单，不来自用户输入。
- 默认不使用静默参数，除非该工具官方推荐且可预览。

安装器调研补充：

| 安装器 | 检测命令 | 安装命令形态 | 权限特点 | v0.3.0 建议 |
|---|---|---|---|---|
| winget | `winget --version` | `winget install --id <id> --exact --source winget` | 部分包会触发 UAC 或安装器交互 | 使用精确 ID，显示 source 和协议确认提示 |
| scoop | `scoop --version` | `scoop install <manifest>` | 默认用户级安装，通常不需要管理员 | 适合开发工具，需提示 bucket 依赖 |
| choco | `choco --version` | `choco install <package> -y` | 默认常需要管理员权限 | 只在明确管理员环境和确认后执行 |

执行策略：

- 检测命令只读取版本和 exit code，可缓存检测结果。
- 安装命令必须使用 `exec.CommandContext` 和超时控制。
- winget 优先使用 `--id`、`--exact`、固定 source，避免模糊搜索安装错包。
- scoop/choco 的包名也必须来自后端白名单。
- 安装日志应流式写入 Wails event，最终结果仍通过 API 返回。

成功标准：

- 能检测本机可用安装器。
- 能安装至少 Node.js / Go / Python / Git / Docker Desktop 中的部分工具。
- 安装过程可取消或至少可明确显示正在执行。
- 安装失败写入 history 并展示 Toast。

## 三、推荐优先级

| 优先级 | 功能 | 原因 |
|---|---|---|
| P0 | 快捷键支持 | 风险低，直接提升效率 |
| P0 | 数据采样与 Dashboard 趋势 | 为自动清理和长期维护提供数据基础 |
| P1 | 自动清理提醒 | 复用 Cleaner 能力，但必须避免静默删除 |
| P1 | 镜像源配置中心 | 用户价值高，但要先做备份和恢复 |
| P2 | 一键安装环境 | 吸引力强，但命令执行和权限风险最高 |

---

## 四、不建议纳入 v0.3.0 的内容

- 后台无确认自动删除文件。
- 任意包名安装或任意命令执行。
- 完整企业级代理策略管理。
- 插件系统。
- 云同步。
- macOS / Linux 完整包管理器集成。

---

## 五、关键风险

1. **命令执行风险**：安装和配置命令必须由后端白名单生成。
2. **配置污染风险**：镜像源和代理必须备份原值，支持恢复。
3. **误删风险**：自动清理只能提醒，不能默认删除。
4. **权限风险**：winget/choco 可能需要管理员权限，UI 必须明确提示。
5. **数据膨胀风险**：图表采样必须限制保留周期。
