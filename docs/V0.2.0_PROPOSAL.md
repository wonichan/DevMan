# DevMan v0.2.0 近期增强方案

> 文档更新时间：2026-05-23
> 目标版本：v0.2.0
> 版本定位：可用性增强 + 扫描/清理能力升级

---

## 一、版本目标

v0.2.0 不再扩展基础骨架，而是把 v0.1.x 已有能力打磨成更接近可日常使用的版本。

核心目标：

1. **操作有反馈**：所有关键异步操作都有 Toast 或确认对话框，避免用户只看到控制台错误。
2. **设置可持久化**：Settings 页面不再只是本地 UI 状态，配置可写入 SQLite。
3. **扫描能力扩大**：支持自定义扫描路径，并新增 Docker / pnpm / yarn / bun 等常见工具识别。
4. **清理能力更实用**：识别大型缓存和依赖目录，给出可选择、可确认的清理建议。
5. **体验入口更高效**：提供全局搜索、基础主题设置和迁移进度可视化能力。

---

## 二、功能范围

### 1. 用户反馈与确认

现状：

- `ToastProvider`、`Toast`、`useToast` 已存在。
- `ConfirmProvider`、`useConfirm` 已存在。
- `Cleaner`、`Migration` 已部分接入。
- `Environments`、`Versions`、部分加载流程仍存在 `console.error`。

方案：

- 保留现有 Toast / Confirm 基础设施。
- 补齐所有页面的错误提示和成功反馈。
- 删除或降级非必要 `console.error`，用户可感知的失败必须走 Toast。
- 所有危险操作统一走 Confirm：迁移、清理、重置数据、删除路径、导入覆盖等。

成功标准：

- 前端业务页面不再把用户可见错误只写入控制台。
- 关键操作成功/失败均有明确提示。
- 危险操作执行前均有二次确认。

---

### 2. Settings 持久化

现状：

- `Settings.tsx` 使用本地 `useState`。
- 后端没有 settings 表和 Settings API。
- `ARCHITECTURE.md` 架构图已预留 settings 表，但实际 schema 未实现。

方案：

新增轻量配置模型：

```go
type AppSettings struct {
    AutoScanOnStartup      bool     `json:"AutoScanOnStartup"`
    ConfirmBeforeMigration bool     `json:"ConfirmBeforeMigration"`
    Theme                  string   `json:"Theme"`
    CustomScanPaths        []string `json:"CustomScanPaths"`
}
```

后端新增：

- SQLite 表：`settings(key TEXT PRIMARY KEY, value_json TEXT, updated_at TIMESTAMP)`
- Registry 方法：`GetSetting` / `SaveSetting` 或 `GetSettings` / `SaveSettings`
- App API：`GetSettings()`、`SaveSettings(settings)`

前端新增：

- `frontend/src/devman-types.ts` 增加 `AppSettings`
- `Settings.tsx` 加载和保存设置
- 保存成功/失败接入 Toast

成功标准：

- 重启应用后设置仍保留。
- Settings 页面初始值来自后端。
- 设置保存失败时用户能看到错误。

---

### 3. 自定义扫描路径

现状：

- `scanner.Engine` 当前依赖 PATH 和固定常见路径。
- Scanner 接口为 `Detect() ([]EnvInstance, []EnvPath, error)`，没有上下文参数。

方案：

短期不大改 Scanner 接口，先在 Engine 层引入扫描选项：

```go
type ScanOptions struct {
    CustomPaths []string
}
```

实现方式：

- `ScanAll()` 从 settings 读取 `CustomScanPaths`。
- Engine 将自定义路径传给支持自定义路径的扫描逻辑。
- 第一版可先覆盖 Node / Python / Java / Go 等安装目录探测。
- Settings 页面支持添加、删除、校验路径。

成功标准：

- 用户添加非标准安装目录后，扫描可识别对应环境。
- 无效路径不会导致扫描失败，只给出用户可见提示或跳过。

### 4. 新增扫描器

优先级建议：

1. **Docker**：独立开发环境，用户价值高。
2. **pnpm / yarn / bun**：Node 生态常用工具，和缓存清理强相关。
3. **VS Code / JetBrains**：体积和缓存可观，但扫描规则更复杂。

后端建议文件：

- `internal/scanner/docker.go`
- `internal/scanner/javascript_tools.go`
- 后续再拆 `ide.go` 或 `jetbrains.go`

需要更新：

- `scanner.NewEngine()` 注册新 scanner。
- `modelsForScanner()` 增加 env metadata。
- scanner 测试覆盖路径检测和版本解析。

成功标准：

- 新工具可在 Dashboard / Environments 中展示。
- 版本号不是占位符。
- 常见缓存路径能进入 `EnvPath`。

---

### 5. Cleaner 深度扫描

现状：

- 已有 `AnalyzeCleanable()` / `CleanItems()` API。
- 清理项模型为 `CleanableItem`，字段较少，但足够第一版使用。

建议第一版识别：

| 类别 | 路径示例 | 默认选择 |
|---|---|---|
| npm cache | `%LocalAppData%\npm-cache` | 是 |
| pnpm store | `%LocalAppData%\pnpm-store` / `.pnpm-store` | 是 |
| yarn cache | `%LocalAppData%\Yarn\Cache` | 是 |
| pip cache | `%LocalAppData%\pip\Cache` | 是 |
| Gradle cache | `%UserProfile%\.gradle\caches` | 是 |
| Cargo registry/cache | `%UserProfile%\.cargo\registry` | 是 |
| Go build cache | `go env GOCACHE` | 是 |
| node_modules | 项目目录下 `node_modules` | 否 |
| Cargo target | 项目目录下 `target` | 否 |

关键原则：

- 缓存类可以默认选择。
- 项目依赖/构建产物不能默认选择。
- 删除前必须二次确认，并展示数量和总大小。

成功标准：

- 清理前能看到每项路径、说明、大小和风险类别。
- 清理后重新分析可更新列表。
- 不删除未选中项目。

---

### 6. 全局搜索

目标：

- 快速跳转页面。
- 搜索环境。
- 搜索操作，例如“扫描”“清理”“迁移”“设置”。

前端建议：

- 新增 `GlobalSearch` 或 `CommandPalette` 组件。
- `App.tsx` 维护全局导航入口。
- 搜索数据来源：静态页面 + `GetEnvs()` 环境列表。

成功标准：

- `Ctrl+K` 或顶部搜索入口可打开搜索。
- 选择结果后可跳转对应页面。
- 搜索环境时可跳转 Environments 或 Migration 相关流程。

---

### 7. 主题切换

现状：

- 项目为深色主题。
- 大量 class 直接使用 `slate` / `emerald` / 硬编码色值。
- Tailwind 配置已有 design token，但页面并未完全 token 化。

建议：

- v0.2.0 先完成主题设置持久化：`dark` / `system`。
- 完整浅色主题不强行塞入第一阶段，避免大规模 UI 重构。
- 后续逐步将硬编码颜色迁移到 CSS 变量或统一 token。

成功标准：

- Settings 中可选择主题策略。
- 当前深色主题不回退。
- 为浅色模式预留结构，但不牺牲稳定性。

---

### 8. 迁移进度实时显示

现状：

- `Migrate()` 是同步调用，完成后一次性返回结果。
- 迁移 UI 当前展示步骤和日志，但没有真实进度事件。

方案：

- 后端 migrator 在关键步骤发出 progress event。
- 前端通过 Wails runtime 订阅事件。
- 第一版按步骤进度，不强求精确字节级进度。

建议事件模型：

```go
type MigrationProgress struct {
    EnvId       int64  `json:"EnvId"`
    Step        string `json:"Step"`
    Message     string `json:"Message"`
    Percent     int    `json:"Percent"`
    BytesMoved  int64  `json:"BytesMoved"`
    TotalBytes  int64  `json:"TotalBytes"`
}
```

成功标准：

- 前端能实时显示当前步骤和进度百分比。
- 迁移失败时保留最后一条错误状态。
- 不影响现有 `Migrate()` 返回结果。

## 三、推荐优先级

| 优先级 | 功能 | 原因 |
|---|---|---|
| P0 | Toast 补齐 + Confirm 统一 | 用户反馈基础，改动小，收益高 |
| P0 | Settings 持久化 | 自定义扫描路径、主题、确认策略都依赖它 |
| P1 | 自定义扫描路径 | 直接提高扫描准确性 |
| P1 | Docker / pnpm / yarn / bun 扫描器 | 扩大识别范围，并服务 Cleaner |
| P1 | Cleaner 深度扫描 | v0.2.0 的主要实用增强 |
| P2 | 全局搜索 | 体验增强，依赖较少 |
| P2 | 主题切换基础 | 完整浅色模式成本高，先做设置结构 |
| P2 | 迁移进度事件 | 后端改造较大，建议放后段 |

---

## 四、不建议纳入 v0.2.0 的内容

- 一键安装环境：依赖 winget / scoop / choco 策略，适合 v0.3.0。
- 网络代理统一配置：涉及多工具配置写入和回滚，适合 v0.3.0。
- 完整浅色主题：当前硬编码颜色较多，建议单独做 UI 主题重构。
- 插件系统：需要稳定 Scanner API 后再设计。
